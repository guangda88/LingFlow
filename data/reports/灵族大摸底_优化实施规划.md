## 一、规划原则

1. **安全第一**: P0 密钥泄露立即处理，不等审批
2. **无破坏优先**: 每步操作必须可逆，删除前备份
3. **验证后执行**: 每项操作完成后运行相关测试确认无回归
4. **分批推进**: 按优先级分 5 个阶段，每阶段独立可执行

---

### Phase 3: 代码重复精简（审批后 7-14 天）

**3-1 self_optimizer 同源演化文件（~200-300 行可合并）**

灵克交叉审计确认：5 组文件是同源独立演化，非性 copy-paste。逐行重复集中在数据结构定义和常量。

- 提取共享数据结构到 `lingflow/self_optimizer/shared_types.py`
- 各项目文件 import 共享部分，保留各自的演化逻辑
- 预估减少 ~200-300 行

**3-2 Result[T] 类型统一（~169 行）**

lingflow、lingminopt、lingyang 各有自己的 Result 实现。

- **方案**: 以 lingflow 的 `Result[T]` 为标准，其他项目 import 或复制
- **风险**: 跨项目依赖（需通过 lingmessage 包分发）
- **提交议事厅决定是否统一**

**3-3 Config 模式（~470 行，概念相似但实现不同）**

概念相同（dict/dataclass/JSON loader），但实现差异大。

- **建议**: 暂不统一。各项目需求不同，强行统一会破坏封装
- **备选**: 提取抽象基类 `BaseConfig` 到 lingmessage 供共享

## 三、风险评估

| 阶段 | 风险等级 | 主要风险 | 缓解措施 |
|------|---------|---------|---------|
| Phase 0 | 🟡 中 | 轮换 Key 后服务中断 | 先生成新 Key，再禁用旧 Key |
| Phase 1 | 🟢 低 | 删除死代码后测试失败 | 全量 pytest + grep 验证 |
| Phase 2 | 🟢 低 | 标注不影响运行 | 只改注释和元数据 |
| Phase 3 | 🟠 中 | 提取共享代码引入新依赖 | 通过 lingmessage 包分发 |
| Phase 4 | 🟢 低 | 文档变更无功能影响 | 纯文档操作 |
| Phase 5 | 🟠 中 | sys.path 清理可能破坏导入 | 逐项目替换，每个替换后跑测试 |
