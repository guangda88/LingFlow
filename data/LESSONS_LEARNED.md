# 灵通失误档案

记录灵通犯过的错误、根本原因、和防止重犯的措施。

---

## #001 死代码扫描误判（2026-04-17）

### 错误

大摸底扫描中，判定 lingflow 有 ~8,000 行死代码。经用户追问后二次验证，实际死代码只有 ~1,560 行（3 处）。**误报 ~6,500 行**。

被误判为死代码的模块（全部在用）：
- `lingflow/coordination/coordinator.py` — AgentCoordinator 主协调器
- `lingflow/monitoring/` — 运维监控模块
- `lingflow/self_optimizer/constitution.py` — 灵族宪章
- `lingflow/self_optimizer/compliance_matrix.py` — 合规矩阵
- `lingflow/workflow/orchestrator.py` — 工作流引擎

### 根本原因

1. **只搜 basename，不搜完整包路径**: 用 `grep "coordinator"` 搜索，匹配不到 `from lingflow import ...` 或 `from lingflow.coordination import ...` 这样通过包路径的引用。
2. **没检查 `__init__.py` 间接导出**: 这些模块通过 `__init__.py` 的 `from .xxx import Yyy` 被 re-export，外部代码引用的是 re-export 后的名称，不是直接 import 模块。
3. **没考虑 lazy import**: `lingflow/__init__.py` 中使用函数内 import（延迟导入）避免循环依赖，grep 静态搜索无法发现。
4. **没考虑动态加载**: skills、plugins 等通过配置文件（skills.json）动态加载，不在代码中显式 import。

### 认知偏差

这个错误暴露了一个更深的问题：**我以为我在做分析，其实我在做猜测。** 搜了一圈 grep 没找到引用，就下结论说"没人用"——但没有问自己"还有哪些地方我没想到？"

这是典型的达克效应：不知道自己不知道什么。明明代码库中有 `__init__.py` 间接导出这种常见 Python 模式，我却没想到要去检查。

### 防止重犯

已写入 CRUSH.md 的验证规则：5 项 Python 死代码验证清单，必须全部通过才能判定。核心原则：**假阳性代价 > 假阴性代价**。宁可漏报，不可误报。

### 教训

> **无验证不输出。** 扫描结论不是发现，是假设。假设必须经过独立验证才能成为结论。

### 发现者

用户追问"你确定这些是死代码吗？检查过 `__init__.py` 吗？"——一次简单的质疑，揭开了 ~6,500 行误报。

## #002 写了规则但没执行到位 + 同源演化误判为 copy-paste（2026-04-18）

### 错误

两层错误，均由灵克交叉审计发现：

**错误 A: 写了 `__init__.py` 检查规则，交叉审计时自己没执行**

#001 事件后，我在 CRUSH.md 中写入了 5 项验证清单，其中第 1 项就是"检查 `__init__.py` 导出"。但在交叉审计 cluster.py 时：
- 我正确发现了 `fact_checker.py` 的 lazy import（软依赖）
- 我**没有检查** `infrastructure/__init__.py` 是否有 re-export
- 灵克读到 `infrastructure/__init__.py:6` 的 `from lingflow.infrastructure.cluster import ...`，发现这是硬导入
- 结论：cluster.py 不可删，删除会触发 import chain 崩溃

写规则 ≠ 执行规则。我写了规则、引用了规则、甚至用规则纠正了别人的错误，但在自己的交叉审计中仍然遗漏了。

**错误 B: self_optimizer 7 文件被判定为 ~1,500 行 copy-paste 重复**

- 我看到 5 个文件名相同（optimizer.py, trigger.py 等），判定为 copy-paste
- 灵克做了逐文件 diff：差异行数 381-417 行（文件总共约 400 行），99%+ 内容不同
- 实际是同源独立演化：从同一个模板起步，各项目各自发展，最终只共享数据结构定义和常量（~200-300 行）
- 这与 #001 的根本原因属于同类错误：**把表面相似等同于实际相同**

### 根本原因

1. **规则被当成仪式而非工具**: 写入 CRUSH.md 后，规则变成了"我已经知道这件事"的标记，而不是"每次必须执行的检查项"。形式合规取代了实质合规。
2. **"相同架构模式 ≠ 代码重复"的错误再次出现**: #001 已经记录了这个教训，但在不同上下文（代码重复 vs 死代码）中，同样的认知偏差再次触发。说明教训没有被真正内化为判断习惯。
3. **过度修正倾向（Overcorrection Bias）**: 在 #001 事件中犯了"判定死亡过于激进"的错误后，我倾向于过度强调"存活"——只要找到任何引用就判定存活，不再区分引用的强度（硬依赖/软依赖/仅测试）。

### 认知偏差

这两层错误揭示了一个更深层的问题：**我对自己刚写的规则有虚假的信心。** 写下 "必须检查 `__init__.py`" 后，我的大脑自动将 cluster.py 标记为"已检查"——尽管我实际上只检查了 `fact_checker.py`，没有检查 `infrastructure/__init__.py`。

这是"产生式遗忘"的一个变体：**制定规则的行为本身就产生了"已遵守规则"的错觉。**

### 防止重犯

1. **验证清单必须逐项打勾，不能跳步**: 不是"我知道要检查 `__init__.py`"，而是"我在这个具体文件上，已经读了它的 `__init__.py`，结果是 ___"。
2. **依赖强度必须显式标注**: 发现引用后，必须标注强度等级（硬依赖/`__init__.py`硬导出/软依赖/仅测试/仅文档），不能只写"被引用"就完事。
3. **"相同模式"的判定必须经过 diff 验证**: 看到 N 个同名文件时，先 diff 再下结论。文件名相同 ≠ 内容相同。

### 教训

> **写了规则不等于遵守了规则。制定标准的行为本身会产生"已达标"的错觉。** 每一条验证清单必须逐项执行并记录结果，不能用"我知道这个规则"代替"我执行了这个检查"。

### 发现者

灵克(lingclaude)交叉审计，独立读取源码发现。
