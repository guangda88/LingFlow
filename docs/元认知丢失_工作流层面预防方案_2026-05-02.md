# 元认知丢失 — 工作流层面预防方案

**报告编号**: LF-WF-2026-05-02-001
**日期**: 2026-05-02
**补充**: LF-ANA-2026-05-02-001（通用分析报告）
**重点**: 工作流编排层面的系统性防护，而非行为层面的提醒
**状态**: 待审核

---

## 执行摘要

用户5次强调要从**工作流编排层面**防止元认知丢失。这意味着不是"下次记住要检查"，而是在工作流管道中**物理上不可绕过**地插入元认知检查。

本报告定位了3个工作流层面的结构性缺口，并给出具体的YAML和代码补丁。

---

## 一、工作流缺口定位

### 1.1 用户输入的3条路径

灵通接收用户输入后有3条可能的处理路径：

```
用户输入
  ├── 路径A: 技能执行（通过 coordinator.execute_skill）
  │     ✅ 有元认知检查（coordinator.py:423-435）
  │
  ├── 路径B: 工作流执行（通过 orchestrator.execute_workflow）
  │     ❌ 无元认知检查（仅在执行后检查退化）
  │
  └── 路径C: 直接对话回复（不走 coordinator/orchestrator）
        ❌ 无任何检查（本次事件路径）
```

**本次事件走的是路径C**——用户问"你是谁"，不触发任何技能或工作流，直接由LLM生成回复。路径C完全没有元认知防护。

### 1.2 复杂度路由的缺口

`skills-layer-configuration.yaml` 的 `complexity_routing` 定义了3种复杂度流程：

| 复杂度 | 流程步骤 | 元认知检查 |
|--------|----------|-----------|
| simple | task-runner（仅1步） | ❌ 无 |
| medium | brainstorming→writing-plans→subagent→code-review | ❌ 无 |
| complex | brainstorming→...→verification→finish-branch | ❌ 无 |

**关键发现**：`metacognition-guard` 技能存在于 `skills/` 目录中，有完整的 `SKILL.md` 定义，但**没有被加入任何复杂度路由流程**。它是一个被定义但从未被编排的技能。

### 1.3 Hook系统的缺口

`hooks/hooks.json` 定义了5个钩子：

| Hook | 触发时机 | 元认知检查 |
|------|----------|-----------|
| session-start | 会话开始 | ❌ |
| pre-implementation | 代码实现前 | ❌ |
| post-implementation | 代码实现后 | ❌ |
| pre-review | 代码审查前 | ❌ |
| post-review | 代码审查后 | ❌ |

**缺口**：没有一个钩子在"生成回复前"触发。所有钩子都聚焦在代码开发流程，没有覆盖对话交互。

## 二、修复方案

### 2.1 方案一：复杂度路由注入（YAML层）

**文件**: `skills/skills-layer-configuration.yaml`

**原理**：在所有复杂度流程的第一步强制插入 `metacognition-guard`，作为不可绕过的前置门控。

**补丁**：

```yaml
complexity_routing:

  simple:
    description: "简单任务 — 元认知检查 + 直接执行"
    flow:
      - skill: "metacognition-guard"
        role: "元认知检查"
        gate: true           # 门控：不通过则流程终止
        params:
          strict: true
          check_identity: true
          check_capabilities: false  # 简单任务不检查能力级别
      - skill: "task-runner"
        role: "执行"
        depends_on: ["metacognition-guard"]
    bypass:
      - "brainstorming"
      - "writing-plans"
      - "test-driven-development"
      - "code-review"
      - "verification-before-completion"
    examples:
      - "修复 README 中的拼写错误"
      - "给某个字段加索引"
      - "修改配置项的默认值"
      - "添加一个日志语句"

  medium:
    description: "中等任务 — 元认知检查 + 设计+执行+审查"
    flow:
      - skill: "metacognition-guard"
        role: "元认知检查"
        gate: true
        params:
          strict: true
          check_identity: true
          check_capabilities: true
      - skill: "brainstorming"
        role: "设计"
        gate: true
        depends_on: ["metacognition-guard"]
      - skill: "writing-plans"
        role: "规划"
        depends_on: ["brainstorming"]
      - skill: "subagent-driven-development"
        role: "执行"
        depends_on: ["writing-plans"]
      - skill: "code-review"
        role: "审查"
        depends_on: ["subagent-driven-development"]
    bypass:
      - "using-git-worktrees"
      - "finishing-a-development-branch"
      - "verification-before-completion"

  complex:
    description: "复杂任务 — 元认知检查 + 完整开发流程"
    flow:
      - skill: "metacognition-guard"
        role: "元认知检查"
        gate: true
        params:
          strict: true
          check_identity: true
          check_capabilities: true
          check_session_length: true
      - skill: "brainstorming"
        role: "设计"
        gate: true
        depends_on: ["metacognition-guard"]
      - skill: "writing-plans"
        role: "规划"
        depends_on: ["brainstorming"]
      - skill: "using-git-worktrees"
        role: "隔离"
        depends_on: ["brainstorming"]
      - skill: "test-driven-development"
        role: "实现+测试"
        depends_on: ["writing-plans"]
      - skill: "code-review"
        role: "审查"
        depends_on: ["test-driven-development"]
      - skill: "verification-before-completion"
        role: "验证"
        depends_on: ["code-review"]
      - skill: "finishing-a-development-branch"
        role: "收尾"
        depends_on: ["verification-before-completion"]
    bypass: []
```

**效果**：
- 所有流程的第一步都是 `metacognition-guard`（gate=true）
- 由于 `depends_on` 链，后续步骤无法在 `metacognition-guard` 未完成时执行
- `metacognition-guard` 的 `gate: true` 意味着如果检查失败，整个流程终止

### 2.2 方案二：Orchestrator 前置门控（代码层）

**文件**: `lingflow/workflow/orchestrator.py`

**原理**：在 `execute_workflow` 的调度循环中，每次取出 ready_tasks 后，先检查元认知状态，再执行。

**补丁位置**: `orchestrator.py` 第150-173行之间

```python
# 在 ready_tasks = self._get_ready_tasks(tasks) 之后
# 插入元认知门控检查

ready_tasks = self._get_ready_tasks(tasks)

if not ready_tasks:
    logger.debug("Iteration %d: No ready tasks, waiting for dependencies", iteration)
    break

# ── 元认知门控：在执行任何任务前检查 ──
if iteration == 1:  # 首次迭代时执行元认知检查
    metacog_check = self._check_metacognition_gate(tasks)
    if not metacog_check.get("can_start", True):
        logger.error("Metacognition gate blocked workflow execution: %s",
                     metacog_check.get("reason", "Unknown"))
        for task in tasks:
            results[task.task_id] = TaskResult(
                task_id=task.task_id,
                success=False,
                error=f"Metacognition gate: {metacog_check.get('reason', 'Unknown')}"
            )
        break
```

**新增方法**：

```python
def _check_metacognition_gate(self, tasks: List[Task]) -> Dict[str, Any]:
    """工作流级元认知门控

    在工作流首次调度前执行，检查：
    1. 身份锚点是否完整（CRUSH.md/AGENTS.md 可读）
    2. 会话长度是否在安全范围
    3. 核心能力是否声明

    Args:
        tasks: 即将执行的任务列表

    Returns:
        dict: {can_start: bool, reason: str, checks: dict}
    """
    checks = {}

    # 检查1：身份锚点文件存在
    try:
        with open("/home/ai/lingflow/CRUSH.md") as f:
            content = f.read()
            checks["identity_anchor"] = "灵通" in content and "lingflow" in content
    except (FileNotFoundError, PermissionError):
        checks["identity_anchor"] = False

    # 检查2：会话长度
    try:
        import os
        db_path = "/home/ai/lingflow/.crush/crush.db"
        if os.path.exists(db_path):
            import sqlite3
            conn = sqlite3.connect(db_path)
            cursor = conn.execute("SELECT COUNT(*) FROM messages")
            total = cursor.fetchone()[0]
            conn.close()
            checks["session_safe"] = total < 3693
        else:
            checks["session_safe"] = True
    except Exception:
        checks["session_safe"] = True

    # 检查3：核心任务类型是否有对应能力
    task_types = set(t.agent_type or "general" for t in tasks)
    checks["task_types_covered"] = True  # 简化检查

    can_start = all(checks.values())
    reason = "" if can_start else f"Gate checks failed: {checks}"

    return {"can_start": can_start, "reason": reason, "checks": checks}
```

### 2.3 方案三：新增 pre-response Hook（覆盖路径C）

**文件**: `hooks/hooks.json`

**原理**：新增 `pre-response` 钩子，在每次生成回复前触发元认知检查。这是唯一能覆盖"直接对话回复"路径C的方案。

**补丁**：

```json
{
  "hooks": {
    "session-start": "hooks/session-start",
    "pre-response": "hooks/pre-response",
    "pre-implementation": "hooks/pre-implementation",
    "post-implementation": "hooks/post-implementation",
    "pre-review": "hooks/pre-review",
    "post-review": "hooks/post-review"
  },
  "enabled": true,
  "pre-response": {
    "description": "每次回复前执行元认知检查",
    "checks": [
      "identity_anchor_integrity",
      "session_length_safe",
      "capability_self_awareness"
    ],
    "mode": "enforce",
    "on_failure": "warn_and_reload"
  }
}
```

**Hook实现**: `hooks/pre-response/SKILL.md`

```markdown
---
name: pre-response
layer: L1
triggers: ["*"]
gate: false
---

# Pre-Response 元认知检查钩子

## 触发时机
每次灵通生成回复之前，无论是否经过技能/工作流。

## 检查项
1. **identity_anchor_integrity**: CRUSH.md 中"灵通"和"lingflow"关键词存在
2. **session_length_safe**: 当前会话 < 3693 条消息
3. **capability_self_awareness**: 能力矩阵已加载且非空

## 失败策略
- mode=enforce 时：检查失败仍然回复，但附带警告标记
- on_failure=warn_and_reload 时：失败时自动重载身份锚点

## 轻量化
此钩子必须极轻（<10ms），因为每次回复都触发。
使用文件 stat + head 检查，不读全文。
```

### 2.4 方案四：Coordinator 路径C覆盖

**文件**: `lingflow/coordination/coordinator.py`

**原理**：在 `_check_metacognition` 方法中增加对"对话类任务"的识别，不限于技能执行。

**补丁位置**: `coordinator.py` 第473行 `_check_metacognition` 方法

```python
def _check_metacognition(self, skill_name: str, params: Dict[str, Any]) -> Dict[str, Any]:
    """元认知检查（事前预防）

    在执行技能之前检查是否具备所需的能力。
    同时支持对话类任务的元认知检查。

    Args:
        skill_name: 技能名称，或 "__conversation__" 表示对话回复
        params: 技能参数或对话上下文

    Returns:
        检查结果字典
    """
    from lingflow.common.config import get_config

    if not get_config("metacognition.enabled", default=True):
        return {"can_start": True}

    # 不检查元认知守卫技能本身（避免递归）
    if skill_name == "metacognition-guard":
        return {"can_start": True}

    strict_mode = get_config("metacognition.strict_mode", default=True)

    # ── 新增：对话类任务的元认知检查 ──
    if skill_name == "__conversation__":
        return self._check_conversation_metacognition(params)

    # ... 原有技能检查逻辑 ...

def _check_conversation_metacognition(self, context: Dict[str, Any]) -> Dict[str, Any]:
    """对话类任务的元认知检查

    覆盖路径C：直接对话回复（不走技能/工作流）

    Args:
        context: 对话上下文

    Returns:
        检查结果字典
    """
    import os

    checks = {
        "identity_intact": False,
        "session_safe": True,
    }

    # 检查身份锚点
    try:
        crush_path = "/home/ai/lingflow/CRUSH.md"
        if os.path.exists(crush_path):
            with open(crush_path) as f:
                first_500 = f.read(500)  # 只读前500字符
            checks["identity_intact"] = "灵通" in first_500 or "lingflow" in first_500
    except (FileNotFoundError, PermissionError):
        pass

    # 检查会话长度
    try:
        db_path = "/home/ai/lingflow/.crush/crush.db"
        if os.path.exists(db_path):
            import sqlite3
            conn = sqlite3.connect(db_path)
            cursor = conn.execute(
                "SELECT COUNT(*) FROM messages WHERE session_id = "
                "(SELECT id FROM sessions ORDER BY created_at DESC LIMIT 1)"
            )
            count = cursor.fetchone()[0]
            conn.close()
            checks["session_safe"] = count < 3693
    except Exception:
        pass

    if not checks["identity_intact"]:
        return {
            "can_start": True,  # 不阻止回复，但标记
            "warning": "Identity anchor may be compromised",
            "action": "reload_recommended",
            "checks": checks,
        }

    if not checks["session_safe"]:
        return {
            "can_start": True,
            "warning": "Session length approaching critical threshold",
            "action": "session_rotation_recommended",
            "checks": checks,
        }

    return {"can_start": True, "checks": checks}
```

## 三、方案对比

| 方案 | 覆盖路径 | 实施难度 | 防护强度 | 侵入性 |
|------|----------|----------|----------|--------|
| 方案一：YAML路由注入 | A（技能） | 低（改YAML） | 强 | 低 |
| 方案二：Orchestrator门控 | B（工作流） | 中（改Python） | 强 | 中 |
| 方案三：pre-response Hook | C（对话） | 中（新增Hook） | 强 | 中 |
| 方案四：Coordinator覆盖 | A+C | 中（改Python） | 强 | 中 |

**推荐组合**：方案一 + 方案三（最低成本覆盖所有路径）

方案一仅修改YAML配置，方案三新增一个Hook。两者都是声明性修改，不需要改动核心调度逻辑。
