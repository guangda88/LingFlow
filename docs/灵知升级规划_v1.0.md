## 数据资产全景

| 层级 | 规模 | 说明 |
|------|------|------|
| **本地原始数据** | ~20TB | 去重前的完整语料（文本、PDF、音频、视频等），不含外连数据集 |
| **云端数据** | ~17.4TB | OpenList索引的7个网盘资源 |
| **外部数据源** | 29个API | arXiv、PubMed、CBETA、CText、Wikisource等，无限量 |
| **去重后本地数据** | 数TB | 去重清洗后的可用语料 |
| **已入库** | 25GB (PostgreSQL) | 结构化+嵌入的数据 |
| **已入库记录** | ~3.5M条 | documents 109K + guoxue 263K + ima 101K + tcm 48K + sys_books 3M |

**总数据资产**: 37TB+（本地20TB + 云端17.4TB）+ 29个外部API数据源
**入库率**: 25GB / 37TB+ ≈ 0.07%。99.93%的数据还在库外。

**九域分布（documents表，已有嵌入）**:

| 域 | 记录数 | 占比 | 评估 |
|----|--------|------|------|
| 中医 | 68,749 | 63% | 最充实 |
| 儒家 | 20,610 | 19% | 较好 |
| 气功 | 15,432 | 14% | 核心域但数据量不足 |
| 佛家 | 3,942 | 4% | 有基础 |
| 科学 | 364 | <1% | 极少 |
| 哲学 | 227 | <1% | 极少 |
| 道家 | 53 | <1% | 极少 |
| 心理学 | 21 | <1% | 极少 |
| 武术 | 8 | <1% | 几乎为空 |

**IMA知识库补充**: 中医73K + 气功18K + 儒家8K + 太极1.6K（尚未生成嵌入）

**核心判断**: 中医和气功是基本盘，但道家、武术、心理学、科学、哲学严重不足。20TB原始数据中可能有大量未提取的补充内容。

灵知不缺工具，缺的是**把20TB数据导入**。

教材处理5步工作流、14个批量导入脚本、BGE-M3嵌入服务、混合检索、OCR标注——全部已构建完成。但20TB原始数据只导入了25GB，入库率0.12%。

## 三阶段规划

### 第一阶段：吃饱 — 数据入库（预计2-3周）

**目标**: 把已有数据全部导入灵知，内容提取率从 <1% 提升到 >60%

#### 1.1 教材批量导入（最高优先级）

| 任务 | 现状 | 目标 | 工具 |
|------|------|------|------|
| 168本TXT教材 | 已导入10本 | 全部导入 | `scripts/import_textbooks.py` |
| 教材TOC解析 | 已构建 | 对168本逐一运行 | `autonomous_processor.py` |
| 教材分块+嵌入 | 已构建 | 全部生成向量 | `generate_block_embeddings.py` |

执行方式：编写批量调度脚本，对168本教材逐一调用已有的5步工作流。每本处理完自动验证，失败的记录原因。

#### 1.2 PDF批量提取

| 任务 | 数量 | 工具 | 备注 |
|------|------|------|------|
| 智能气功教材PDF | ~100本 | `content_extraction/extractor.py` | 双引擎（PyMuPDF+pdfplumber） |
| 辅导材料PDF | ~50本 | 同上 | |
| 杂志扫描版 | 1992-1999 | OCR标注器 | 需要OCR，扫描质量参差 |

优先级：教材 > 辅导材料 > 杂志。扫描版杂志放最后，因为OCR质量需要人工校对。

#### 1.3 古籍数据导入

| 任务 | 数量 | 工具 |
|------|------|------|
| CBETA佛典 | 已有本地XML | `scripts/import_cbeta.py` |
| CText道家典籍 | 在线爬取 | `scripts/import_daoist_ctext.py` |
| Wikisource道家 | 在线爬取 | `scripts/import_daoist_wikisource.py` |
| 国学数据库 | 263K条已有 | 确认嵌入完整 |

这些脚本都已就绪，执行即可。

#### 1.4 云端资源评估

17.4TB云端数据（通过OpenList可达）。先做分类盘点：

- 哪些是灵知九域相关的内容
- 哪些格式可以直接提取（TXT、PDF）
- 哪些需要转录（音频、视频）
- 按领域优先级排序，制定分批导入计划

#### 第一阶段验收标准

- [ ] 168本教材全部导入并生成嵌入
- [ ] PDF教材提取率 >80%
- [ ] 古籍数据导入完成
- [ ] documents 表记录数 > 50万
- [ ] 向量覆盖率 > 80%

### 第二阶段：消化 — 质量提升（预计2-3周，与第一阶段可部分并行）

**目标**: 检索质量达到实用水平，为下游应用提供可靠的知识支撑

#### 2.1 RAG管线优化

| 环节 | 现状 | 优化方向 |
|------|------|---------|
| 嵌入模型 | BGE-M3 (512维) | 评估是否需要升级维度或模型 |
| 分块策略 | 通用分块 | 按领域调整（古籍用句号分块，教材按章节分块） |
| 混合检索 | 已实现RRF融合 | 调参：向量/BM25权重比 |
| Reranker | BGE-reranker-v2-m3 | 确认部署状态，评估精排效果 |
| 检索评估 | 无系统评估 | 建立领域测试集，量化召回率和准确率 |

#### 2.2 领域知识结构化

灵知九域的数据不应只是"文本块+向量"。每个领域有独特的结构：

| 领域 | 结构化方向 | 已有基础 |
|------|-----------|---------|
| 气功 | 功法→步骤→要点→禁忌 | `qigong/content_parser.py` |
| 中医 | 症状→方剂→药材→医案 | `tcm_documents` 表 |
| 武术 | 拳种→套路→动作→要领 | 待建 |
| 古籍 | 典籍→篇章→句读→注释 | `guoxue_content` 表 |
| 儒/道/佛 | 经典→注释→流变 | `domains/` 模块 |

#### 2.3 检索反馈闭环

已有 `knowledge_gaps` 和 `feedback` API。接入实际使用场景，收集"查不到"和"查不准"的案例，反哺数据导入优先级。

#### 第二阶段验收标准

- [ ] 建立九域检索测试集（每域至少50个问答对）
- [ ] 混合检索 Top-10 召回率 > 85%
- [ ] RAG回答引用准确率 > 80%
- [ ] 知识缺口自动识别上线

### 第三阶段：看见 — 视觉接入（第一阶段完成后启动）

**目标**: 视觉传感器与灵知知识库对接，实现"看着你，告诉你"

#### 3.0 硬件现实

**边缘设备**: Luckfox Lyra Ultra (RK3506B)
- 三核 ARM Cortex-A7 (1.2GHz) + Cortex-M0
- 512MB DDR3L, 8GB eMMC
- SC3336 300万像素摄像头
- **无NPU**，无RKNN支持

**结论**: 这台设备是**图像采集终端**，不是推理终端。512MB内存跑不动任何有意义的神经网络模型。

**正确架构**:

```
Luckfox (采集) → 网络传输 → 服务端 (3090/本机CPU) → 灵知API (知识匹配) → 返回结果
  拍照/录像        JPEG/MP4      姿态估计/OCR/分类       RAG检索          结构化结果
  基础预处理
```

设备端只做：拍照、录像、压缩、传输。所有推理在服务端完成。

**GPU现状**: 3090目前运行 Fish2 (语音合成)。需要协调GPU时间片，或确认Fish2运行间隙是否可以跑推理任务。

#### 3.1 边缘设备部署（明天硬件到货后）

| 任务 | 说明 | 难度 |
|------|------|------|
| 系统启动 | 烧录系统、SSH连接、网络配置 | 低 |
| 摄像头驱动 | SC3336适配、测试拍照 | 中 |
| 图像采集服务 | 定时拍照/按需拍照、JPEG压缩、HTTP上传 | 中 |
| 服务端推理服务 | 3090上部署姿态估计/OCR/分类模型 | 高 |
| 通信协议 | 设备 → 灵知API（HTTP上传图像 → 返回分析结果） | 中 |

#### 3.2 古籍OCR（最优先的视觉应用）

为什么最优先：古籍数字化是知识库扩容的关键路径，大量古籍只有扫描版。
OCR不需要边缘设备，可以直接在服务端跑。

| 步骤 | 说明 | 运行位置 |
|------|------|---------|
| OCR引擎选型 | PaddleOCR（中文最优）/ TrOCR / 现有Tesseract评估 | 服务端 |
| 批量扫描处理 | 对PDF/DJVU古籍批量OCR | 服务端（3090或CPU） |
| 人工校对流程 | OCR结果 → 灵知标注系统 → 人工校对 | 灵知Web UI |
| 校对结果入库 | 校对后文本导入documents表 | 灵知服务端 |
| OCR模型微调 | 用校对数据微调OCR模型 | 3090 |

**边缘设备的OCR场景**: 拍摄纸质古籍 → 上传服务端OCR → 返回识别结果。适合古籍收藏机构实地数字化。

#### 3.3 中医视觉辅助（第二优先）

| 应用 | 流程 | 对照知识 |
|------|------|---------|
| 面诊 | 边缘拍照 → 服务端面部特征提取 → 灵知检索 | 中医面诊文献 |
| 舌诊 | 边缘拍照 → 服务端舌象分割分类 → 灵知检索 | 舌诊图谱 |
| 手诊 | 边缘拍照 → 服务端掌纹/掌色分析 → 灵知检索 | 手诊文献 |

**重要约束**: 中医视觉辅助定位为**参考工具**，不是诊断工具。输出必须包含免责声明。

#### 3.4 姿态识别应用

| 应用 | 服务端模型 | 对照知识 |
|------|-----------|---------|
| 气功站桩姿势 | MediaPipe/YOLO-Pose → 关键点 → 标准姿态对比 | 教材中的动作要领 |
| 八段锦教学 | 动作序列识别 → 实时反馈 | 功法步骤描述 |
| 太极拳套路 | 全身姿态跟踪 | 套路分解动作 |
| 康复训练 | 姿态评估 → 纠正建议 | 康复训练标准动作库 |

**边缘设备角色**: 拍摄视频流 → 关键帧提取 → 上传服务端 → 返回姿态分析结果。实时性取决于网络延迟和服务端推理速度。

#### 第三阶段验收标准

- [ ] Luckfox Lyra Ultra 拍照并上传图像到灵知API
- [ ] 服务端OCR跑通，古籍OCR准确率 > 90%（通用文本）> 80%（古文）
- [ ] 至少一个姿态识别应用端到端可用（拍照→分析→返回）
- [ ] 灵知API支持图像输入
