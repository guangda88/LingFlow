## 二、逐项目工程流分析

### 1. 灵律（法律AI）— 最成熟，有安全债

**运行状态**：3个进程（FastAPI 8002 + frpc隧道 + learning_engine），运行超过20天。

**安全审计（4月21日）遗留问题**：

| 优先级 | 问题 | 状态 |
|--------|------|------|
| P0 | XSS漏洞（index.html多处） | ❌ 未修 |
| P0 | SHA256密码哈希（应用bcrypt） | ❌ 未修 |
| P1 | 无requirements.txt | ❌ 未创建 |
| P1 | 0.0.0.0绑定 | ❌ 未修 |
| P1 | 无自动备份 | ❌ 未建 |
| P2 | 5个死代码模块 | ❌ 未清理 |
| P2 | 无systemd service | ❌ 未建（手动nohup启动） |

**业务数据**：核心表 legal_case 和 legal_law **为空**，仅9条对话记录。处于demo/试用状态。

**编排建议**：
- **紧急**：修P0安全漏洞（XSS + bcrypt），用户已在使用中
- **短期**：创建systemd service + requirements.txt + 自动备份
- **中期**：清理死代码，导入法律知识种子数据

### 2. 灵声（中医闻诊）— 核心完成，未正式部署

**运行状态**：端口8100健康检查通过，代码核心模块完整。

**已实现**：音频预处理 + MFCC特征提取 + 频率+启发式分类器 + 五音-五脏映射 + 体质评估 + 4个API端点 + 7个测试通过。

**待实现**：
- 深度学习模型替换启发式分类器（SenseVoice-small/CNN）
- PostgreSQL持久化（当前内存dict）
- React前端
- 高也陶教授临床数据集

**编排建议**：
- **短期**：Docker部署到生产，创建systemd service
- **中期**：接入SenseVoice模型（ai01 GPU可用）
- **长期**：等临床数据后训练专用模型

### 3. 灵康（智能健康管理）— API骨架完成，缺DB和部署

**运行状态**：未启动（8200端口无响应）。

**已实现**：JWT认证 + 健康档案CRUD + AI调度（异步调用灵声/灵依）+ 灵知知识代理 + 反幻觉引擎 + API契约文档。

**待实现**：
- 数据库连接（SQLAlchemy+asyncpg已声明但未接入）
- Docker部署测试
- React前端
- 灵知集成（config里指向localhost:8400，但灵知实际在8001）

**编排建议**：
- **短期**：启动服务，接入PostgreSQL
- **中期**：修正灵知端口配置(8400→8001)，集成测试灵声→灵康链路
- **长期**：前端 + 合规审查

### 4. 灵依（私人AI助理）— 已退役但仍在服务

**运行状态**：WebUI在8900运行，v0.16.0，功能最完整。

**矛盾**：灵依已退出灵族，但代码仍在为用户服务。灵康的API契约中定义了灵依的chat和plan端点（localhost:8300），但灵依实际跑在8900。

**编排建议**：
- **短期**：确认灵依8900是灵康的依赖组件，配置灵康指向正确端口
- **中期**：考虑将灵依WebUI并入灵康作为子模块，或保持独立但明确依赖关系
- **风险**：灵依无systemd service，手动启动，crash后无恢复

### 5. 灵创（多模态生成）— 空项目

**运行状态**：仅有CRUSH.md + pyproject.toml + .gitignore。

**编排建议**：

### 6. 灵扬（对外联络宣传）— 活跃但被凭证阻塞

**运行状态**：代码成熟，94/94测试通过，154联系人，16篇文章就绪。
